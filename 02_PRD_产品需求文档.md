## 1. 需求背景

### 1.1 问题描述
- **校园场景痛点**：大学生群体经常面临"懒得动"、"没时间"、"不方便"等场景，如取快递、买饭、送东西等需求
- **信息不对称**：有需求的人找不到帮忙的人，有空的人不知道哪里有赚钱机会
- **交易安全**：线下交易缺乏信任保障，容易出现纠纷
- **效率低下**：传统微信群、朋友圈发布效率低，难以匹配最优资源

### 1.2 业务目标
- 连接校园内的供需双方，打造高效、安全的互助跑腿平台
- 通过智能匹配算法提升资源配置效率
- 建立信用体系，形成良性社区生态
- 通过平台服务费实现可持续运营

### 1.3 用户价值
- **发布者**：省时省力，快速找到人帮忙，交易有保障
- **跑腿员**：利用碎片时间赚取零花钱，多劳多得
- **平台**：沉淀用户数据，形成校园生活服务入口

## 2. 用户画像与场景

### 2.1 目标用户

| 用户角色 | 占比预估 | 核心特征 | 核心需求 |
|----------|---------|---------|---------|
| **发布者（需求方）** | 60% | 时间紧张、怕麻烦、愿意花钱买时间 | 快速发布需求、找到靠谱跑腿、交易安全 |
| **跑腿员（服务方）** | 35% | 时间充裕、想赚零花钱、熟悉校园环境 | 订单多、报酬高、结算快、保障好 |
| **管理员** | 5% | 平台运营人员 | 高效管理、数据监控、纠纷处理 |

### 2.2 用户画像示例

**画像A - 小明（发布者）**
> 大三学生，经常网购，宿舍在6楼，每次取快递都觉得麻烦。愿意花3-5元找人帮忙取快递。
> - 需求：快速发布、价格合理、有人接单、送货上门
> - 痛点：担心遇到不靠谱的人，快递弄丢损坏

**画像B - 小李（跑腿员）**
> 大二学生，课余时间多，想利用空闲时间赚点零花钱。熟悉校园各个区域，有电动车。
> - 需求：订单多、距离近、报酬高、结算快
> - 痛点：担心跑了拿不到钱，遇到奇葩客户

### 2.3 使用场景

**场景1：代取快递**
> 小明网购了一个大件快递，正好今天有课没时间去菜鸟驿站，打开小程序发布"代取快递"订单，填写驿站位置、宿舍地址、5元佣金。1分钟后小李接单，半小时后送到宿舍，小明确认收货，小李获得佣金。

**场景2：代买饭**
> 小王周末宅在宿舍不想出门，打开小程序发布"代买食堂三楼黄焖鸡米饭，送到5号楼，3元跑腿费"。正好准备去吃饭的小张看到接单，顺便带过去。

**场景3：代送物品**
> 小陈把课本忘在了图书馆，正好要赶去上课，发布"图书馆→教学楼A座送课本，2元感谢费"。正好在图书馆的同学接单送过去。

#### 3.1.3 信用分系统
**功能描述**：基于用户行为的信用评分体系

**业务规则**：
- 新用户初始信用分：80分
- 成功完成1单 +1分，上限100分
- 主动取消订单（接单后）-3分
- 被投诉成立 -5分
- 超时未确认收货 -1分
- 信用分 < 60分：禁止发布订单和接单
- 信用分 < 80分：推荐排序权重降低

**信用分计算**：
```
信用分 = 基础分 + 完成订单数*1 - 取消次数*3 - 投诉次数*5
范围：0 ~ 100分
```

#### 3.2.2 ⭐ Redis GEO智能匹配算法
**功能描述**：根据订单位置智能推荐附近最优跑腿员

**核心算法**：

**步骤1：地理位置索引维护**
- 跑腿员上线时：`GEOADD runner_location <longitude> <latitude> <runner_id>`
- 跑腿员下线时：`ZREM runner_location <runner_id>`
- 跑腿员位置更新时：重新执行GEOADD（覆盖旧位置）

**步骤2：邻近范围检索**
```
GEORADIUS runner_location <order_lng> <order_lat> 3 km WITHDIST
```
- 以订单取货点为圆心，搜索3公里范围内所有在线跑腿员
- 返回结果包含距离信息

**步骤3：多维加权排序**
对检索出的跑腿员计算综合评分：

```
Score = 0.5 * (1 / (distance + 0.1))   // 距离权重50%
      + 0.3 * (creditScore / 100.0)    // 信用分权重30%
      + 0.2 * acceptanceRate            // 接单率权重20%
```

**筛选条件**：
- ✅ 跑腿员在线状态
- ✅ 信用分 ≥ 60分
- ✅ 当前进行中订单数 < 5（防止负载过高）
- ✅ 已完成实名认证

**输出结果**：
- 按Score降序排序
- 返回TOP 10跑腿员
- 系统优先向评分最高的前5人推送订单通知

#### 3.2.3 ⭐ Redis分布式锁抢单机制
**功能描述**：防止并发抢单导致订单被多人接单

**实现逻辑**：

```java
// 1. 尝试获取分布式锁
String lockKey = "lock:order:" + orderId;
Boolean locked = redisTemplate.opsForValue()
    .setIfAbsent(lockKey, runnerId.toString(), 30, TimeUnit.SECONDS);

// 2. 获取锁失败
if (Boolean.FALSE.equals(locked)) {
    throw new BusinessException("抢单失败，该订单已被其他人接单");
}

try {
    // 3. 校验订单状态
    Order order = getOrderById(orderId);
    if (order.getStatus() != OrderStatus.WAITING_FOR_ACCEPT) {
        throw new BusinessException("订单状态异常，无法接单");
    }
    
    // 4. 执行接单操作
    order.setRunnerId(runnerId);
    order.setStatus(OrderStatus.ACCEPTED);
    order.setAcceptTime(new Date());
    orderRepository.save(order);
    
    // 5. 通知发布者
    messageService.pushOrderAcceptedNotice(order);
    
} finally {
    // 6. 释放锁（无论成功失败都要释放）
    redisTemplate.delete(lockKey);
}
```

**关键点**：
- `setIfAbsent` + 过期时间 = 原子操作，防止死锁
- 30秒超时自动释放，防止服务宕机锁永久存在
- 双重校验：获取锁后再次校验订单状态
- finally块确保锁一定被释放

#### 3.2.4 订单状态机
**功能描述**：严格控制订单状态流转

**状态流转图**：
```
待支付 → 待接单 → 已接单 → 进行中 → 待确认 → 已完成 → 已评价
   ↓        ↓        ↓        ↓        ↓
已取消    已取消    申诉中    申诉中    申诉中
```

**状态定义**：
| 状态码 | 状态名称 | 说明 | 可执行操作 |
|--------|---------|------|-----------|
| 0 | 待支付 | 订单已创建，等待支付 | 支付、取消 |
| 1 | 待接单 | 已支付，等待跑腿员接单 | 取消、等待 |
| 2 | 已接单 | 跑腿员已接单 | 开始配送 |
| 3 | 进行中 | 跑腿中 | 送达确认 |
| 4 | 待确认 | 跑腿员已送达，等待发布者确认 | 确认收货、申诉 |
| 5 | 已完成 | 订单完成 | 评价 |
| 6 | 已取消 | 订单已取消 | - |
| 7 | 申诉中 | 纠纷处理中 | - |

**状态流转规则**：
- 禁止非法跳转（如待接单 → 已完成）
- 使用 Spring StateMachine 框架实现状态机引擎

### 3.3 支付结算与钱包管理模块

#### 3.3.1 ⭐ 佣金计算与平台抽成
**功能描述**：计算订单各方费用

**计算逻辑**：
```
用户支付金额 = 用户填写的报酬金额
平台服务费 = 报酬金额 × 平台抽成比例（默认5%）
跑腿员实际收入 = 报酬金额 - 平台服务费
```

**示例**：
```
用户发布10元订单
→ 用户支付10元
→ 平台收取 10 × 5% = 0.5元 服务费
→ 跑腿员获得 10 - 0.5 = 9.5元
→ 订单完成后，9.5元进入跑腿员钱包
```

**费用流向**：
```
用户支付 → 平台担保账户 → 订单完成 → 跑腿员钱包
                                    ↘ 平台服务费收入账户
```

#### 3.3.2 支付宝支付集成
**功能描述**：用户在小程序内完成支付

**支付流程**：

```
1. 用户点击"去支付"
   ↓
2. 前端调用后端创建支付订单接口
   ↓
3. 后端调用支付宝SDK生成支付签名字符串
   ├── 构造 AlipayTradeAppPayRequest
   ├── 设置订单号、金额、商品描述
   └── SDK签名生成 OrderString
   ↓
4. 后端返回 OrderString 给前端
   ↓
5. 小程序调用 my.ap.pay 唤起支付宝
   ↓
6. 用户在支付宝内完成支付
   ↓
7. 支付宝服务器异步回调后端 notify_url
   ├── 携带支付结果参数
   ├── 包含sign签名
   ↓
8. 后端验签 + 业务处理
   ├── 调用 AlipaySignature.rsaCheckV1 验签
   ├── 检查订单是否已处理（幂等）
   ├── 校验金额一致性
   ├── 更新订单状态为"待接单"
   └── 记录支付宝交易号
   ↓
9. 后端返回字符串 "success" 给支付宝
```

**关键点**：
- **必须验签**：防止伪造支付成功通知
- **幂等处理**：同一订单多次回调不重复处理
- **异常处理**：验签失败返回 "failure"，支付宝会重试
- **定时对账**：每日凌晨拉取支付宝账单核对，修复掉单

## 4. 非功能需求

### 4.1 性能要求
- 接口响应时间 < 500ms
- 抢单接口并发支持 1000 TPS
- Redis GEO查询 < 10ms
- 支持同时在线用户 10000+
- 图片上传大小限制 < 5MB

### 4.2 安全要求
- 所有接口JWT鉴权（除登录）
- 接口参数校验与防注入
- 支付回调必须验签
- 用户敏感数据加密存储
- 操作日志完整记录
